# trading_bot — пакет пайплайна

Ниже — подробное описание модуля **расчёта уровней по горизонтальному объёму** (пики сглаженного volume profile, HVN), как он устроен в репозитории.

---

## Уровни горизонтального объёма (HVN): цель и место в системе

**Задача:** по истории минутных свечей выделить ценовые зоны с повышенной концентрацией объёма (узловые уровни), ранжировать их по «силе», сохранить в SQLite и дальше использовать в аналитике касаний (`level_events`) и в выгрузке Google Sheets.

**Важно:** это не «классический» фиксированный VP за один календарный день на графике терминала. Здесь профиль строится по **выбранному окну 1m-данных** из базы (календарный месяц или lookback от последней свечи — см. ниже), бины идут по цене, объём суммируется по свечам, попадающим в бин.

---

## Файлы и поток данных

| Этап | Где |
|------|-----|
| Ядро алгоритма | `trading_bot/analytics/volume_profile_peaks.py` |
| Адаптация параметров под монету | `get_adaptive_params()` в том же файле |
| Пересчёт и запись в БД | `trading_bot/scripts/rebuild_volume_profile_peaks_to_db.py` |
| INSERT в `price_levels` | `trading_bot/data/volume_profile_peaks_db.py` |
| Планировщик (суточный прогон) | `trading_bot/data/scheduler.py` — задача ~02:45 UTC |
| Потребители активных уровней | `trading_bot/analytics/level_events.py` (только `is_active=1`, `level_type='volume_profile_peaks'`), `trading_bot/entrypoints/export_to_sheets.py` |

**Источник OHLCV:** таблица `ohlcv`, таймфрейм **`1m`**, спот Binance (как хранится в БД). Для расчёта используются колонки **`open, high, low, close, volume, timestamp`**.

---

## Шаг 1. Окно данных (rebuild)

Скрипт `rebuild_volume_profile_peaks_to_db.py` для каждого символа из `TRADING_SYMBOLS` (см. `trading_bot/config/symbols.py`) формирует датафрейм `work`:

- Если заданы **`PRO_LEVELS_LOOKBACK_DAYS`** и/или **`PRO_LEVELS_LOOKBACK_HOURS`** в `trading_bot/config/settings.py` — окно от **последней 1m-свечи в БД** назад на эту длительность.
- Иначе — **предыдущий календарный месяц UTC** (срез через `slice_calendar_month_utc` из `dynamic_accumulation_zones`).

Итоговый набор свечей передаётся в `find_pro_levels()`. Строка **`layer`** в БД кодирует окно (например `volpeak_{days}d_{hours}h_{start}_{end}` или аналог для месяца).

---

## Шаг 2. Построение профиля объёма по цене

Внутри `find_pro_levels()`:

1. Берётся последняя цена **`current_price`** (последний `close` в окне).
2. **Шаг бинирования профиля** `tick_size_eff`:
   - если в параметрах передан явный `tick_size` — он используется;
   - иначе эвристика **`max(current_price * 0.0005, 1e-8)`** (половина «типа 0.05%» от цены, но не ноль).
3. Каждая свеча относится к бину: **`price_bin = round(close / tick_size) * tick_size`**.
4. **Профиль:** сумма `volume` по каждому `price_bin` → ряд «цена → объём».
5. **Сглаживание:** скользящее среднее по этому ряду (`rolling`, окно по умолчанию **5**, нечётное), получается `volume_sm` — по нему ищутся «пики».

**Округление цены в ответе для человека/БД:** опционально по **биржевому tick** (Binance spot → запасной вариант Bybit `instruments`) через `symbol=` — на **построение бинов профиля** это не влияет, только на число знаков в поле `Price`.

---

## Шаг 3. Пороги по «высоте» пика (strong / weak)

На сглаженном профиле `volume_sm`:

- Задаются квантили и множители к среднему (см. `height_percentile`, `height_percentile_strong`, `height_percentile_weak`, опционально `height_mult`).
- Строятся подмножества **strong** и **weak** — бины, где сглаженный объём выше соответствующих порогов.

Пороги могут быть **адаптивными** (`get_adaptive_params`): например `height_mult` завязан на отношение квантиля объёма к среднему; также считаются вспомогательные метрики (волатильность, CV объёма) для `distance_pct`, `valley_threshold`, динамического `dynamic_merge_pct` и т.д.

Переопределения из кода: константы **`PRO_LEVELS_*`** в `settings.py` (часть из них подмешивается в `rebuild` в `_compute_find_pro_params`).

---

## Шаг 4. Склейка соседних пиков по «долинам» (valley merge)

Функция **`merge_by_valley`**: соседние по цене пики объединяются в кластер, если «долина» между ними на сглаженном профиле **неглубокая** относительно минимума из двух пиков (порог `valley_merge_threshold` / `valley_threshold`), либо если пики **очень близки** по цене (`merge_distance_pct`).

Итог кластера — суммарный объём на участке, **POC** — цена бина с максимальным объёмом внутри кластера.

На этапе **мягкого** прохода двухпроходного режима valley-merge для профиля может быть отключён (`enable_valley_merge=False`), чтобы не дублировать агрессивную склейку.

---

## Шаг 5. Жадный отбор уровней (`greedy_level_selection`)

По отфильтрованным бинам strong/weak:

- Для каждого кандидата задаётся **ценовой коридор** вокруг цены бина: **`± distance_pct`** (внутри прохода часто используется уменьшенная доля, например половина от основного `distance_pct`).
- В коридор попадают свечи, **пересекающие** `[lower, upper]` по **`high/low`** (не только `close` внутри — явно для учёта «пилы»).
- **Минимальная длительность** присутствия цены в коридоре — `min_duration_hours` (через число свечей × медианный шаг времени).
- **Скор (score):** по умолчанию нормализованный **`volume × duration`**, кандидаты сортируются по убыванию score.
- **Жадно** выбирается до `top_n` / с учётом `max_levels`, без перекрытия: второй уровень не берётся, если его цена ближе чем `distance_pct` к уже выбранному.
- Опционально **`reserved_prices` + `exclude_reserved_pct`:** не брать кандидатов слишком близко к уже зарезервированным ценам (используется во втором проходе).

После отбора для каждой строки ещё раз считается **снимок полосы** (`_snapshot_band_volume`): суммарный объём, **POC** как бин по `close` с максимальной суммой объёма внутри полосы, длительность и интервал времени.

---

## Шаг 6. Тиры (Tier 1 / 2 / 3)

По **длительности** нахождения цены в зоне (в часах), с порогами по умолчанию **48 ч** и **12 ч** (или `duration_thresholds`):

- **Tier 1 (Бетон)** — выше первого порога;
- **Tier 2 (Сильный)** — между первым и вторым;
- **Tier 3 (Локальный)** — ниже второго.

Флаг **`include_all_tiers`:** если `False`, слабый тир может отбрасываться ещё на этапе одного прохода.

---

## Шаг 7. Дедупликация и финальный merge

1. **`_dedup_level_rows`:** близкие по цене строки складываются в один «ведро» с шагом, зависящим от `tick_size`, `current_price` и `dedup_round_pct`; остаётся строка с большим объёмом (при равенстве — большая длительность).

2. **`_resolve_final_merge_pct`:** порог слияния соседних итоговых уровней по относительному расстоянию цены — либо задан явно, либо оценивается по медианным зазорам между ценами, с **клампом** в диапазоне порядка **3–5%** (`FINAL_MERGE_PCT_CLAMP_MIN/MAX`), чтобы итог не был «лесенкой» из уровней в доли процента.

3. **`_merge_close_level_rows` + `_cluster_level_row_groups`:** соседние уровни объединяются в группы с **якорем** (не цепочка «каждый к предыдущему» — чтобы не схлопнуть весь диапазон транзитивно). Дополнительно может действовать **`final_merge_valley_threshold`:** слияние разрешено только если «долина» на сглаженном профиле между якорями не слишком глубокая относительно пиков.

После merge полоса пересчитывается объединённым коридором **`± distance_pct`** от min/max цен группы, tier 1 из группы может сохраняться через `preserve_tier1_from_group`.

---

## Шаг 8. Двухпроходный режим (по умолчанию)

`two_pass_mode=True` (если не включён `PRO_LEVELS_LEGACY_WEAK_MERGE`):

1. **Жёсткий проход (strict):** более строгие пороги для weak-части (`strict_height_percentile_weak`, `strict_height_mult`). Результат после dedup/merge формирует **резерв цен** (`strict_full`).

2. **Tier 1 только из жёсткого финала:** для строк с Tier 1 выполняется уточнение полосой `± distance_pct` с принудительным **`tier_override=TIER1_LABEL`**, чтобы «бетон» не дробился после merge.

3. **Мягкий проход (soft):** более низкие пороги (`soft_height_percentile_*`, `soft_height_mult`), больший `soft_min_duration_hours` по умолчанию, кандидаты **исключаются** вблизи цен резерва (`reserved_prices`, `exclude_reserved_pct`). Уровни мягкого прохода **не получают Tier 1** (потолок Tier 2), чтобы не дублировать «бетон».

4. **Объединение strict Tier 1 + soft**, затем **ещё одна** финальная склейка с тем же классом merge и сохранением Tier 1 внутри группы.

Если `run_soft_pass=False`, возвращаются только уточнённые Tier 1 из жёсткого прохода.

---

## Шаг 9. Сохранение в `price_levels`

`save_volume_profile_peaks_levels_to_db`:

- Деактивирует прежние активные строки для пары **`symbol` + `level_type='volume_profile_peaks'`**.
- Вставляет новые строки: цена, объём пика, длительность, тир, `start_utc`/`end_utc` → в т.ч. `t_start_unix` / `t_end_unix`, поле **`layer`**, `strength` (сейчас как proxy = объём), `is_active=1`.

История старых расчётов остаётся в таблице с `is_active=0`.

---

## Связь с `level_events`

`level_events` читает **активные** `volume_profile_peaks` и метрики из `instruments` (ATR, tick). Логика касаний завязана на **1m Binance** и **дневной ATR**. Без заполненных уровней и ATR событий может не быть.

---

## Настройка без правки алгоритма

См. **`trading_bot/config/settings.py`**: блок **`PRO_LEVELS_*`** (lookback, пороги, merge, soft/strict, `PRO_LEVELS_RUN_SOFT_PASS`, и т.д.). Значение `None` у большинства полей означает «взять из адаптивных параметров или дефолт в коде».

---

## Краткая схема пайплайна

```text
SQLite ohlcv (1m)
       → окно (месяц или lookback)
       → бины по close, суммарный volume
       → сглаживание
       → пороги strong/weak, valley merge
       → greedy selection по коридорам high/low
       → тиры по длительности
       → dedup → финальный merge с клампом %
       → [двухпроходно: strict → soft с резервом]
       → price_levels (volume_profile_peaks)
       → level_events / Google Sheets
```

---

## Отличие от `dynamic_accumulation_zones`

В проекте есть отдельный пайплайн **динамических зон накопления** (`trading_bot/analytics/dynamic_accumulation_zones.py`, используется в части логики экспорта). HVN в `volume_profile_peaks.py` — **отдельная** ветка: пики сглаженного профиля, жадный отбор и двухпроходная схема «бетон + остаток». Для продуктовых решений важно не смешивать эти два подхода без явной необходимости.

## HTF-уровни (`volume_profile_htf`)

Отдельный сценарий для **крупного таймфрейма** и **длинного окна в днях** (по умолчанию **1d** и **365 дней**). Используется тот же `find_pro_levels`, но вход — не 1m, а выбранный ТФ из `ohlcv`; в БД пишется **`level_type = volume_profile_htf`**, существующий расчёт **`volume_profile_peaks` не затрагивается**.

- **Код:** `trading_bot/analytics/htf_volume_levels.py`, запуск: `python -m trading_bot.scripts.run_htf_volume_levels` (из корня репозитория с `PYTHONPATH` на корень).
- **Символы:** `TRADING_SYMBOLS` → источник Binance; `ANALYTIC_SYMBOLS["macro"]` → yfinance; `ANALYTIC_SYMBOLS["indices"]` → TradingView (как в коллекторах).
- **Настройки:** блок `HTF_LEVELS_*` в `trading_bot/config/settings.py` и одноимённые переменные окружения (`HTF_LEVELS_LOOKBACK_DAYS`, `HTF_LEVELS_TIMEFRAME`, пороги тиров, `HTF_LEVELS_TOP_N`, …).
- **Google Sheets:** лист по умолчанию `htf_volume_levels`. Отдельная таблица: задать `HTF_LEVELS_SHEET_ID` или `HTF_LEVELS_SHEET_URL` (или `HTF_LEVELS_SHEET_TITLE`); иначе используется та же книга, что и у основного аудита (`MARKET_AUDIT_SHEET_*`). Отключить выгрузку: `HTF_LEVELS_DISABLE_SHEETS=1` или флаг `--no-sheets`.
- **Проверка данных:** `python -m trading_bot.scripts.run_htf_volume_levels --check-data` — по каждому символу считает строки `ohlcv` в окне с ожидаемым `source` (`binance` / `yfinance` / `tradingview`) и без фильтра; если при нуле с `source` есть строки «как угодно» — в логе подсказка о несовпадении источника в БД.
- **Догрузка под HTF:** `python -m trading_bot.scripts.backfill_ohlcv_for_htf` — спот `1d`, макро `1d,1w,1M` (без `4h`, см. комментарий в скрипте), индексы; опции `--spot-timeframes`, `--macro-timeframes`, `--skip-spot`, `--skip-macro`, `--skip-indices`.
- **Эксперимент (4h, ~3 мес, отдельный лист и тип в БД):** сначала догрузить `4h` (спот + при необходимости `update_indices()`), затем  
  `python -m trading_bot.scripts.run_htf_volume_levels --timeframe 4h --lookback-days 90 --level-type volume_profile_htf_4h_90d --worksheet htf_4h_3m`

`level_events` и стандартный `export_to_sheets.py` **не подхватывают** `volume_profile_htf` автоматически — это отдельный слой для отчётов/ручного анализа, пока вы явно не свяжете его с пайплайном событий.
