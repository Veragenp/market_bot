# Спецификация: структурный старт цикла и групповой триггер (финальная версия для реализации)

Документ фиксирует логику отбора «двух сильных зон», согласованности пула по \(W_i\), mid-полосы, триггеров по WebSocket и отмены по коллективному пробою.  
Текущий `cycle_levels` (медианная зона от live `ref_price`) **не отменяется** до миграции: новый модуль может идти параллельно под флагом.

---

## 1. Проблема

Живая цена как единственный якорь для eligibility даёт «плавающие ворота»: один и тот же уровень то проходит, то нет без смены структуры.  
Нужен старт и точка отсчёта, привязанные к **паре сильных зон** и **согласованности пула** в ATR-нормализации.

---

## 2. Согласованность пула: не корреляция Пирсона

Имеется в виду **одномерная согласованность** массива \(W_i\) (ширина коридора в ATR по каждой монете).  
**Выброс** — монета, у которой \(W_i\) сильно отличается от типичного уровня пула.

### Рекомендуемый способ: MAD вокруг медианы

Для пула порядка **~20 монет** и редких «диких» альтов удобнее **MAD**, чем IQR:

- **Медиана** устойчива к выбросам (в отличие от среднего).
- **MAD** — робастный масштаб разброса; при 1–2 выбросах не «ломает» границы так, как могут квартили на малой выборке после фильтров.

**Формулы:**

- \(m = \mathrm{median}(\{W_i\})\)
- \(\mathrm{MAD} = \mathrm{median}(\{|W_i - m|\})\)
- Выброс, если \(|W_i - m| > k \cdot \mathrm{MAD}\), по умолчанию **\(k = 3\)**

**IQR** (Q1, Q3, правило 1.5×IQR) оставить как **опциональный второй фильтр** или альтернативу в A/B, если захочешь сравнить на логах.

## 3. Алгоритм (рабочая версия)

### Шаг 1. Кандидаты и пара (L, U) на монету

- Берём активные уровни из `price_levels` (как минимум `vp_local`, опционально `manual_global_hvn` — whitelist как в `cycle_levels`).
- Относительно **текущей цены WebSocket (last)**:
  - кандидаты **ниже** цены;
  - кандидаты **выше** цены.
- **Не менее N** кандидатов с каждой стороны (по умолчанию **N=1**: достаточно одного ниже и одного выше ref для пары L/U; параметр `STRUCTURAL_MIN_CANDIDATES_PER_SIDE`). Иначе монета: `incomplete_structure`.
- Сортировка кандидатов **детерминированная**: `volume_peak` DESC, затем `strength` DESC, затем `updated_at` DESC.
- **Пара уровней:** из **топ-K** по объёму с каждой стороны (K = 5 по умолчанию). Режим выбора пары — **настраиваемый параметр**:
  - **Режим A:** strongest ниже + strongest выше (первые в отсортированном списке каждой стороны среди топ-K).
  - **Режим B:** ближайший значимый ниже + ближайший значимый выше (порог значимости — отдельный параметр, TBD).
- Опционально **толщина уровня** для дальнейших касаний/пробоев: ±**0.1 ATR** к цене уровня (настраиваемо).

### Шаг 2. Ширина коридора и фильтр пула

- \(\mathrm{ATR}_i\) — дневной ATR монеты из `instruments.atr` (как сейчас в проекте).
- \(W_i = (U_i - L_i) / \mathrm{ATR}_i\) (всё в цене монеты, нормализация своим ATR).
- По всем монетам с валидной парой считаем \(m\), \(\mathrm{MAD}\), помечаем выбросы при \(k=3\).
- Действие для выброса (настраиваемо): переподбор пары внутри топ-K → иначе **исключить монету** из пула.
- Если после фильтра монет **меньше минимума** для торговли — сценарий паузы / уведомление / математический fallback (**TBD**, зафиксировать числом перед продом).

### Шаг 3. Mid-полоса (точка отсчёта на монету)

- \(\mathrm{mid}_i = (L_i + U_i) / 2\) (цена).
- Ширина полосы: **±P% от ширины зоны в цене** \((U_i - L_i)\), не в ATR.  
  Пример: P = 2% → полоса \([\mathrm{mid} - 0.02\cdot(U-L),\ \mathrm{mid} + 0.02\cdot(U-L)]\).

### Шаг 4. Триггеры (WebSocket)

- **Старт таймера входа:** если **≥ N_touch** монет (например 5) **коснулись** своей mid-полосы в **реалтайме** (last trade внутри полосы).  
  Нужны: источник цены (last), **дебаунс** (не считать каждый тик отдельным событием; например не чаще 1 события «касание» на символ в X секунд — настраиваемо).
- **Окно для N_touch** (опционально): например все касания в течение 15 минут — параметр.
- После срабатывания — **таймер** (например 5 мин) и далее логика **отбоя** от нижней зоны (шорт) / верхней (лонг) — детали исполнения в спеке исполнения / tutorial_v3.

### Шаг 5. Отмена сценария (коллективный пробой)

- Если **≥ N_abort** монет (например 5) показали **пробой против ожидаемого отбоя**:
  - цена ушла от **нижнего** выбранного уровня **вниз** на ≥ **D_abort** ATR (например 0.3), **или**
  - от **верхнего** уровня **вверх** на ≥ **D_abort** ATR.
- Тогда: **стоп сценария**, новый поиск пары / другой диапазон / перезапуск алгоритма (настраиваемо).
- Границы уровня согласовать с буфером шага 1 (±0.1 ATR).

## 5. Таблицы БД (SQLite, предложение)

### 5.4 Связь с существующими таблицами

- **`price_levels`:** `level_below_id` / `level_above_id` — явная трассировка выбранных HVN.
- **`instruments.atr`:** источник ATR на шаге расчёта (снимок в `structural_cycle_symbols.atr`).
- **`trading_state`:** фазы торгового цикла (`arming`, `in_position`, `closed`, …), `cycle_id`, связь с исполнителем (tutorial_v3).

### 5.5 Единственный writer для `cycle_levels`

После внедрения structural-пайплайна **запись в `cycle_levels` (и смена торгового `cycle_id` в `trading_state`) делает только structural-слой** в момент **ARMED / freeze** (передача снимка в исполнение).

- Плановый `rebuild_cycle_levels()` по таймеру **не должен** перетирать снимок без согласования (отключить, вызывать только из orchestrator, или оставить для режима «legacy»).
- Исполнитель (**tutorial_v3**) **только читает** `cycle_levels` + `trading_state`.

### 5.6 Исполнение и возврат к structural после сделки

**Контракт:** `cycle_levels` — что торгуем на цикл; исполнитель — вход/выход по своим правилам.

Пример ветки **LONG после передачи данных:**

1. Вход в LONG.
2. Исходы:
   - **Тейк до зоны противоположной стороны** (в терминах снимка — уровень/зона **SHORT**): по смыслу переход к сценарию **противоположного направления**; нужен **пересчёт** актуальных уровней выхода / следующего входа (тот же «пересчётчик», что готовит снимок для цикла).
   - **Стоп:** аналогично — **пересчёт** и далее по алгоритму (новый ARMED или новая фаза structural — зафиксировать в машине состояний исполнителя).

3. После **закрытия** сценария (тейк, стоп, отмена по правилам) управление снова у **structural-слоя**: новый проход сканирования / ожидание N_touch / новый ARMED — как в §3.

Детали порядка вызовов (один ли `commit_cycle_snapshot` и на тейк, и на стоп) — **TBD** в спеке исполнителя; на уровне данных повторный freeze снова идёт через **единственный writer**.
