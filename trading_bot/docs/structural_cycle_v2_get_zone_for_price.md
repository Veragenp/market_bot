# `get_zone_for_price` в `structural_cycle_v2`: подробный разбор

## Где находится

- Модуль: `trading_bot/analytics/structural_cycle_v2.py`
- Метод: `MarketSyncDetector.get_zone_for_price(price, channels, atr) -> Optional[str]`

Метод возвращает одну из зон:

- `'low'`
- `'mid'`
- `'high'`
- `None` (если зона не может быть определена)

---

## Назначение метода

`get_zone_for_price` определяет, в какой рыночной зоне находится текущая цена для символа относительно набора валидных каналов.

Важная особенность текущей версии:

1. Сначала метод пытается найти канал, куда цена попадает с учетом расширения по ATR (`tolerance`).
2. Если ни один канал не подошел, метод принудительно выбирает ближайший канал и возвращает край (`low`/`high`) по положению цены.

Это устраняет "пустые" зоны в ситуациях, где цена слегка вне диапазона всех каналов.

---

## Как формируются `channels`: `get_channels_for_symbol`

Перед тем как вызвать `get_zone_for_price`, модуль строит каналы через:

- `MarketSyncDetector.get_channels_for_symbol(levels, atr) -> List[Channel]`

Это ключевой шаг, потому что именно он задает геометрию диапазонов, относительно которых потом определяется `low/mid/high`.

## Входные параметры

## `levels: Sequence[Level]`

Список уровней символа. Каждый уровень содержит:

- `price`
- `strength`
- `level_id` (опционально)

Внутри метода уровни дополнительно сортируются по цене (`ascending`), даже если на входе пришли уже отсортированные.

## `atr: float`

ATR символа. Используется для нормализации ширины канала:

`width_atr = (high - low) / atr`

## Алгоритм `get_channels_for_symbol`

## Шаг 0. Валидация

Если:

- уровней меньше двух, или
- `atr <= 0`,

метод возвращает пустой список `[]`.

## Шаг 1. Сортировка уровней

`levels` сортируются по `price` от меньшего к большему.

## Шаг 2. Построение пар соседних уровней

Метод перебирает только соседние пары:

- `(level[i], level[i+1])`

То есть не рассматриваются "дальние" комбинации (`i` и `i+2`, `i+3`, ...).

## Шаг 3. Фильтр по ширине в ATR

Для каждой соседней пары считается:

- `width = high.price - low.price`
- `width_atr = width / atr`

Пара становится каналом, только если:

- `min_width_atr <= width_atr <= max_width_atr`

Где `min_width_atr` и `max_width_atr` — параметры детектора.

## Шаг 4. Формирование зон канала

Для принятой пары вычисляется допуск:

- `tolerance = zone_tolerance_atr * atr`

И сохраняются границы:

- `low_zone_high = low_level.price + tolerance`
- `high_zone_low = high_level.price - tolerance`

Эти поля записываются в `Channel` и могут использоваться в аналитике/отладке, хотя в текущей версии `get_zone_for_price` пересчитывает tolerance заново из переданного ATR.

## Что это означает по смыслу

- Каналы строятся только на локальной структуре (соседние уровни).
- Слишком узкие и слишком широкие каналы отбрасываются по ATR-норме.
- Получаем список "рабочих" локальных коридоров, по которым затем классифицируется цена в `get_zone_for_price`.

---

## Входные параметры

## `price: float`

Текущая цена символа.

## `channels: List[Channel]`

Список валидных каналов для символа, собранных заранее в `get_channels_for_symbol`.

Каждый `Channel` содержит:

- `low_level.price`
- `high_level.price`
- `width_atr`

и служебные поля.

## `atr: float`

ATR символа (из БД), используемый для вычисления допуска:

`tolerance = zone_tolerance_atr * atr`

---

## Алгоритм (шаг за шагом)

## Шаг 0. Базовая валидация

Если нет каналов или `atr <= 0`, метод сразу возвращает `None`.

Причина:

- без каналов нечего классифицировать;
- без ATR невозможно корректно рассчитать `tolerance`.

## Шаг 1. Расчет допуска

Вычисляется:

`tolerance = self.zone_tolerance_atr * atr`

Если, например:

- `zone_tolerance_atr = 0.2`
- `atr = 100`

то `tolerance = 20`.

## Шаг 2. Поиск канала, куда цена попадает в расширенный диапазон

Для каждого канала строится расширенная рамка:

- `low_ext = low_level.price - tolerance`
- `high_ext = high_level.price + tolerance`

Если `low_ext <= price <= high_ext`, цена считается относящейся к этому каналу, и дальше определяется зона:

- если `price <= low_level.price + tolerance` -> `'low'`
- если `price >= high_level.price - tolerance` -> `'high'`
- иначе -> `'mid'`

То есть внутри одного канала зоны определяются как:

- нижняя полоса (с допуском) = `low`
- верхняя полоса (с допуском) = `high`
- центральная часть между ними = `mid`

## Шаг 3. Fallback: ближайший канал

Если на шаге 2 не найден ни один подходящий канал:

- ищется ближайший канал по расстоянию до его границ;
- затем возвращается:
  - `'low'`, если цена ниже `best_ch.low_level.price`
  - `'high'` во всех остальных случаях

Это гарантирует детерминированный результат, даже когда цена вне всех расширенных диапазонов.

---

## Почему может быть выбран только первый подходящий канал

На шаге 2 используется последовательный проход `for ch in channels`, и метод возвращает зону сразу при первом совпадении.

Практический смысл:

- поведение детерминировано;
- нет дополнительных правил выбора "лучшего" канала среди нескольких пересекающихся.

Если каналов много и они могут перекрываться, порядок списка `channels` влияет на итог.

## Примеры

## Пример 1: цена в середине канала

- `low=100`, `high=200`, `atr=50`, `zone_tolerance_atr=0.2`
- `tolerance=10`
- расширенный диапазон: `[90, 210]`
- `price=150`

Результат: `'mid'`.

## Пример 2: цена около нижней границы

Те же параметры, `price=108`.

- `price <= low + tolerance` -> `108 <= 110` истинно

Результат: `'low'`.

## Пример 3: цена выше всех каналов

Допустим, ни один канал не содержит цену даже с расширением.

Срабатывает fallback:

- находится ближайший канал;
- цена выше его `high_level.price`.

Результат: `'high'`.

## Пример 4: невалидный ATR

- `atr <= 0`

Результат: `None`.

---

## Связь с `compute_distribution`

В `compute_distribution` метод вызывается так:

- строятся каналы через `get_channels_for_symbol(levels, atr)`
- затем зона определяется через `get_zone_for_price(price, channels, atr)`

Если вернулся `None`, символ идет в `invalid`.  
Иначе счетчик увеличивается в `low/mid/high`.

Итоговая статистика затем используется в `get_synced_direction` для определения `long/short/None`.

---

## Текущее поведение по смыслу

- Метод "мягко" классифицирует цену в канале с ATR-допуском.
- Если цена вне всех каналов, он все равно дает полезную крайовую классификацию (`low/high`) через ближайший канал.
- Благодаря этому уменьшается доля `invalid` и повышается устойчивость групповой синхронизации на шумных движениях цены.
